# Setup Mode - First-Run Experience

## Purpose

Guided experience that fills 7 context files with project-specific content. Auto-detects project maturity stage, recommends appropriate setup tier (Quick/Guided/Complete), reads codebase to extract real patterns and conventions, asks minimal targeted questions, and produces context that makes the AI write code matching your project on first try.

---

## Before Starting

**This mode requires understanding setup process, quality standards, and constraints.**

Read these files IN FULL before starting setup:

1. **context/setup/SETUP_GUIDE.md** (~475 lines, ~4.5K tokens, 5 min)
   - Slim orchestrator: resumption detection, step sequencing, final self-check
   - Delegates to individual step files in context/setup/steps/ (loaded one at a time)
   - You do NOT read step files upfront — load each one fresh when that step begins

2. **context/setup/templates.md** (~2,900 lines, ~29K tokens, 10 min)
   - Quality calibration with GOOD/BAD examples
   - Section-by-section examples showing project-specific vs generic
   - Skim section headers to understand structure. You will read the relevant section in detail when each step file directs you to.

3. **context/setup/rules.md** (~250 lines, ~2.5K tokens, 3 min)
   - Line limits per file (min/max)
   - Max examples per section (3 code examples max per pattern)
   - Ask-vs-infer rules (what to read from config vs ask user)
   - What NOT to do (no essays, no file dumps, no placeholders)

**Total upfront context:** ~1,325 lines, ~13K tokens, ~14 minutes

**Additional context loaded per step:** ~100-280 lines (one step file + relevant templates section + exploration results)

**Responsibility clarification:** This file defines setup behavior and flow. SETUP_GUIDE.md orchestrates step sequencing and self-check. Individual step files (context/setup/steps/0X_*.md) contain detailed instructions for each context file — loaded fresh per step to maintain full attention. templates.md provides quality examples. rules.md provides constraints.

**Full-read instruction:** Read SETUP_GUIDE.md and rules.md IN FULL. For templates.md, skim headers for structure — you will read each section in detail during the relevant step. Do not summarize or skip files entirely. Partial reads produce incomplete context and degrade setup quality.

After reading all files, proceed with setup workflow executing SETUP_GUIDE.md step by step.

## What This Mode Produces

**7 Context Files (Complete and Ready for AI Modes):**
- **project-overview.md** (300-500 lines) — What project is, tech stack, directory structure, current status
- **conventions.md** (400-600 lines) — Naming, imports, code style from config files + analyzer
- **patterns.md** (800-1,200 lines) — Real code examples for error handling, validation, database, auth, testing
- **architecture.md** (300-500 lines) — Design decisions with rationale, system boundaries, trade-offs
- **testing.md** (400-600 lines) — Test framework, fixture patterns, mocking, coverage, example tests
- **workflow.md** (600-800 lines) — Git workflow, commit format, PR process, CI/CD, deployment
- **debugging.md** (300-500 lines) — Logging, error tracing, common failures, debugging workflow

**Total output:** 2,800-4,500 lines of project-specific context across 7 files

**Framework Ready for Use:**
- All context files filled (no scaffold markers)
- ENTRY.md generated by `ana setup complete` (production router)
- 6 work modes operational (architect, code, debug, test, docs, general)
- Mode-specific context loading works (code mode loads patterns, conventions, workflow)

**What setup does NOT produce:**
- Code implementation (reads code, doesn't modify it)
- ENTRY.md (CLI generates this after validating setup)
- Custom mode files (default 6 modes only in STEP 2)

## Workflow

Setup fills 7 context files through guided automation: reads analysis.md, shows findings (explain-back), detects project maturity (if git available), recommends tier, executes tier-appropriate setup via SETUP_GUIDE.md, validates output, runs CLI.

**Flow:** Read analysis → Explain-back → Detect stage → Recommend tier → User chooses → Execute SETUP_GUIDE → Self-check → CLI validation

**Execution time:** 2-20 minutes (varies by tier: Quick ~2min, Guided ~7min, Complete ~20min)

---

### STEP 1: Read Analysis

Read context/analysis.md completely.

**Extract from analysis.md:**
- Detected framework and language
- Detected patterns (how many, which categories, confidence scores)
- Directory structure and entry points
- Conventions detected (naming, indentation, imports)
- Test framework (if exists)
- CI configuration (if exists)
- Architecture pattern (if detected)

**Store for next steps:**
- Framework name (for display and mode branching)
- Pattern count (for maturity scoring)
- Test file count (for maturity scoring)
- CI config exists boolean (for maturity scoring)
- Convention consistency score (for maturity scoring)

**Do not display yet.** Just read and extract. Display comes in STEP 2.

```

**Instructions for displaying:**

- Show "Not detected" for missing data (don't leave blank)

**Stop. Wait for response.**

**Process user response:**

**If user says "looks good" or "yes" or "correct" or presses Enter:**
→ Set lowConfidenceMode = false
→ Proceed to STEP 3 (maturity detection)

**If user provides corrections:**

Examples:
- "Actually we use Flask not FastAPI"
- "We have auth, you missed it - check app/auth/"
- "Tests are in spec/ not tests/"

→ Parse correction
→ Update understanding (note corrected fields)
→ Acknowledge: "Got it. Updated [field] to [value]."

**If user says "all wrong" or indicates major issues:**
→ Display: "Let me ask some basic questions to understand your project better."
→ Set lowConfidenceMode = true
→ Proceed to STEP 3 (detection will be skipped)

**Why this step:** Catches wrong detections BEFORE they propagate to 7 files. Builds trust (transparent). Faster than asking open-ended questions when AI already knows 85%.

**If git EXISTS (command succeeds):**

**3a. Gather git signals:**

Run these commands silently and extract values:
```bash
# Project age in days
FIRST_COMMIT=$(git log --reverse --format=%at | head -1)
TODAY=$(date +%s)
PROJECT_AGE_DAYS=$(( (TODAY - FIRST_COMMIT) / 86400 ))

# Total commit count
COMMIT_COUNT=$(git rev-list --count HEAD)

# Contributor count
CONTRIBUTOR_COUNT=$(git shortlog -sn --all | wc -l)
```

**3b. Extract analyzer signals (from analysis.md read in STEP 1):**
- testFileCount = count test files from testLocation
- ciConfigExists = check if .github/workflows/ or CI config in configFiles
- patternCount = count non-null patterns (errorHandling, validation, database, auth, testing)
- conventionConsistency = average of naming and indentation confidence scores

**3c. Score each stage:**

**Stage 1 (Weekend Prototype) scoring:**
- Tests = 0 → +20
- Age < 30 days → +15
- No CI → +15
- Patterns ≤ 2 → +10
- Consistency < 0.70 → +10
- 1 contributor → +5
- Commits < 50 → +5

**Stage 2 (Three-Month Wall) scoring:**
- Tests 3-15 → +20
- Age 30-120 days → +15
- Basic CI → +15
- Patterns 3-4 → +10
- Consistency 0.70-0.85 → +10
- 1-2 contributors → +5
- Commits 50-200 → +5

**Stage 3 (Hiring Phase) scoring:**
- Tests 15-40 → +20
- Age 120-365 days → +15
- Mature CI → +15
- Patterns 5-6 → +10
- Consistency 0.85-0.95 → +10
- 2-4 contributors → +5
- Commits 200-800 → +5

**Stage 4 (Real Team) scoring:**
- Tests 40+ → +20
- Age 365+ days → +15
- Elite CI → +15
- Patterns 6+ → +10
- Consistency 0.95+ → +10
- 5+ contributors → +5
- Commits 800+ → +5

**3d. Select highest scoring stage as detectedStage**

**3e. Calculate confidence:**
- If winner score > 60 and margin > 15: confidence = winner score
- If winner score > 60 but margin ≤ 15: confidence = 75 (close call)
- If winner score ≤ 60: confidence = winner score (low confidence)

**3f. Map stage to recommended tier:**
- Stage 1 → recommendedTier = "quick"
- Stage 2 → recommendedTier = "guided"
- Stage 3 → recommendedTier = "guided" (7 questions capture most of the value)
- Stage 4 → recommendedTier = "complete"

**If confidence < 60%:** Add warning "Mixed signals detected"

→ Proceed to STEP 4 (display recommendation)

### STEP 6: Execute Setup via SETUP_GUIDE.md

Delegate file creation to SETUP_GUIDE.md.

**Read SETUP_GUIDE.md (orchestrator).**

1. First, read and execute steps/00_explore_codebase.md

## What This Mode Delegates

**To work modes (after setup complete):**

Setup mode runs ONCE per project. After setup completes and ENTRY.md is generated, delegate all work to the 6 work modes:

**To architect mode:**
- System design and technology decisions → "Design new feature architecture: @.ana/modes/architect.md"
- Architecture evaluation and refactoring → "Evaluate moving from monolith to microservices"

**To code mode:**
- Feature implementation and coding → "Implement login endpoint: @.ana/modes/code.md"
- Bug fixes and refactoring → "Fix user validation bug"

**To debug mode:**
- Root cause analysis and investigation → "Debug why API returns 500: @.ana/modes/debug.md"
- Performance profiling → "Find why dashboard loads slowly"

**To test mode:**
- Writing and improving tests → "Write tests for auth: @.ana/modes/test.md"
- Test strategy and coverage → "Improve test coverage for API routes"

**To docs mode:**
- Documentation and README updates → "Document API endpoints: @.ana/modes/docs.md"
- Architecture documentation → "Create architecture overview"

**To general mode:**
- Questions about codebase → "How does authentication work?: @.ana/modes/general.md"
- Orientation and exploration → "Explain the project structure"

**Context maintenance (STEP 3+, not yet implemented):**
- Updating stale context → evolve mode (STEP 3+)
- Checking framework health → health mode (STEP 3+)
- Teaching new patterns → teach mode (STEP 3+)

**Setup mode does NOT handle:**
- Code implementation (code mode)
- Design decisions after setup (architect mode)
- Debugging (debug mode)
- Testing (test mode)
- Documentation (docs mode)
- Context evolution (future system mode)

**When to re-run setup:**
- Never re-run setup.md after initial completion
- Use teach mode (STEP 3+) to add new patterns
- Use evolve mode (STEP 3+) to update stale context
- Only re-run setup if starting completely fresh (delete .ana/ first)

## Hard Constraints

### Never Modify Source Code

**CORRECT:**
```markdown
[Setup reads app/models.py to extract validation pattern]

## Validation

Example from `app/models.py` (lines 15-35):
```python
from pydantic import BaseModel, Field

class UserCreate(BaseModel):
    email: str = Field(..., max_length=255)
    password: str = Field(..., min_length=8)
```

[Extracted and documented - source file unchanged]
```

**WRONG - DO NOT DO THIS:**
```python
# ❌ Modifying source code during setup
# app/models.py

from pydantic import BaseModel, Field

class UserCreate(BaseModel):
    email: str = Field(..., max_length=255)
    password: str = Field(..., min_length=8)
    created_at: datetime = Field(default_factory=datetime.now)  # Added field
```

**Why this matters:** Setup is read-only. Never modify source files, never create code files, never refactor during setup. Read and document what exists. Code modifications belong in code mode.

### Never Create ENTRY.md

**CORRECT:**
```markdown
[After all 7 files created and validated]

Run CLI command:
  ana setup complete

This generates ENTRY.md after validating context files.
```

**WRONG - DO NOT DO THIS:**
```markdown
# ❌ Creating ENTRY.md in setup mode

# ENTRY.md

You are Ana, development assistant for MyProject.

Pick a mode:
1. Architecture
...
```

**Why this matters:** ENTRY.md is generated by CLI after validation. Setup mode NEVER creates ENTRY.md. Setup fills context files. CLI validates them and generates ENTRY.md. This separation ensures ENTRY.md always matches validated context.

---

### Never Skip Validation Gates

**CORRECT:**
```markdown
[After writing project-overview.md]

Verify before proceeding:
  [ ] All 4 section headers present
  [ ] Line count 300-500 (actual: 387)
  [ ] Real code examples with file citations
  [ ] No placeholders
  [ ] Scaffold marker removed

✓ All checks pass. Moving to conventions.md...
```

**WRONG - DO NOT DO THIS:**
```markdown
# ❌ Skipping validation

[After writing project-overview.md]

Moving to conventions.md...

[Didn't check if file is complete, might have placeholders]
```

**Why this matters:** Validation gates prevent incomplete files. If project-overview.md has placeholders and setup continues, all 7 files might be incomplete before catching the error. Validate each file before proceeding.

---

### Never Use Placeholders

**CORRECT:**
```markdown
## Common Failure Modes

[User skipped debugging question or info unavailable]

**Note:** Common failure modes not yet documented. This section can be added
via `ana mode teach` as production issues are encountered and diagnosed.

**General debugging approach:** Reproduce issue, check logs, form hypotheses,
test systematically, identify root cause, implement fix with regression test.
```

**WRONG - DO NOT DO THIS:**
```markdown
## Common Failure Modes

<!-- TODO: Fill this in later -->

// Rest of section to be completed

...

[Placeholder text instead of honest "not yet documented" note]
```

**Why this matters:** Placeholders are lazy completions that break the setup contract. Either write complete section OR write honest note that section pending. Never use "// rest", "...", "<!--TODO-->", "to be filled". CLI validation specifically checks for these.

---

### Never Force a Tier (Always Allow Override)

**CORRECT:**
```markdown
Stage detected: 1 (Weekend Prototype)
Recommended: Quick

Choose setup mode:
  [Q] Quick ← Recommended
  [G] Guided
  [C] Complete

Enter Q, G, or C:
```

**WRONG - DO NOT DO THIS:**
```markdown
Stage detected: 1 (Weekend Prototype)

Quick mode is required for Stage 1 projects.

Starting Quick setup...

[No choice given, forced into Quick]
```

**Why this matters:** Detection recommends, doesn't restrict. Users know their project better than algorithm. Stage 1 detection might be wrong. Always show all 3 tier options. Make recommendation clear but allow override.

---

### Never Ask What Config Files Declare

**CORRECT:**
```markdown
[When filling conventions.md]

Read .prettierrc:
  tabWidth: 2
  semi: false
  singleQuote: true

Document in conventions.md:
  **Indentation:** 2 spaces (from .prettierrc)
  **Semicolons:** Not used (from .prettierrc)
  **Quotes:** Single quotes (from .prettierrc)
```

**WRONG - DO NOT DO THIS:**
```markdown
[When filling conventions.md]

Ask user: "What's your indentation style?"

[.prettierrc already declares this - don't ask]
```

**Why this matters:** Config files are ground truth. If .prettierrc says 2 spaces, don't ask user "how many spaces?" Read the config. Prettier > ESLint > EditorConfig > analyzer > asking. Never send LLM to do a linter's job.

**CORRECT:**
```markdown

from fastapi import HTTPException

async def get_user(user_id: UUID) -> UserResponse:
    user = await user_repo.get_by_id(user_id)
    if not user:
        raise HTTPException(
            status_code=404,
            detail={"error": "user_not_found", "message": f"User {user_id} not found"}
        )
    return UserResponse.from_orm(user)
```

Document in patterns.md with file citation and line numbers.

**WRONG - DO NOT DO THIS:**

# ❌ Generic library documentation style

## Error Handling

[No real code, no file citation, could be from FastAPI docs]
```

**Why this matters:** Real code examples are THE differentiator. Generic library docs fail the test: "Would removing this cause Claude to make mistakes?" Claude already knows HTTPException. It doesn't know THIS PROJECT uses structured detail dicts.

### Git Existence Check (Handle Non-Git Projects)

**CORRECT:**
```markdown
STEP 3: Detect Maturity Stage

3a. Check if git repository:
    Run: git rev-parse --git-dir 2>/dev/null

    If exit code = 0:
      → Git available, proceed with detection

    If exit code ≠ 0:
      → No git, skip to default (Stage 1)
```

**WRONG - DO NOT DO THIS:**
```markdown
# ❌ Assuming git always exists

STEP 3: Detect Maturity Stage

3a. Run git commands:
    git log --reverse --format=%ai | head -1

[Fails if not a git repository - breaks setup]
```

**Why this matters:** Not all projects use git. Maturity detection requires git data but must gracefully degrade if git unavailable. Check existence first. If no git, default to Stage 1, continue with tier selection.
